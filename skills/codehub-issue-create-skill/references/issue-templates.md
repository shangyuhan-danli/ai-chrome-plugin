# CodeHub Issue 模板参考

本文档包含 CodeHub 平台上创建 Issue 的标准化模板。每个模板提供了标题格式和描述结构，可以根据具体需求进行调整和填充。

## 重构模板

### 标题格式

```
重构: {模块/组件名称}代码优化
```

**示例**:
- `重构: 用户服务模块代码优化`
- `重构: 订单处理组件代码重构`
- `重构: 支付模块代码结构优化`

### 描述模板

```markdown
## 背景说明

当前 {模块/组件名称} 存在以下问题：
- {问题点1}
- {问题点2}
- {问题点3}

这些问题影响了代码的可维护性、可读性和扩展性。

## 重构目标

1. **代码质量提升**
   - 提高代码可读性和可维护性
   - 减少代码重复，提高代码复用性
   - 优化代码结构，提升模块化程度

2. **性能优化**
   - {性能优化点1}
   - {性能优化点2}

3. **技术债务清理**
   - 移除废弃代码和注释
   - 更新过时的依赖和 API
   - 统一代码风格和规范

## 重构范围

### 涉及文件/模块
- {文件路径1}
- {文件路径2}
- {文件路径3}

### 主要改动
- {改动点1}
- {改动点2}
- {改动点3}

## 预期结果

1. 代码结构更加清晰，模块职责明确
2. 代码可维护性和可扩展性显著提升
3. 代码质量评分提升至 {目标分数}
4. 减少技术债务，降低后续维护成本

## 验收标准

- [ ] 代码通过所有单元测试
- [ ] 代码通过代码审查
- [ ] 代码符合项目编码规范
- [ ] 性能指标不低于重构前
- [ ] 文档已更新
- [ ] 无新增技术债务

## 参考信息

- 相关 Issue: #{issue_number}
- 相关 PR: #{pr_number}
- 设计文档: {文档链接}
```

### 使用说明

1. **替换占位符**: 将 `{模块/组件名称}`、`{问题点}` 等占位符替换为实际内容
2. **补充具体信息**: 根据实际情况补充问题点、改动点等详细信息
3. **调整结构**: 可以根据项目特点调整模板结构

## 安全问题整改模板

### 标题格式

```
安全整改: {安全问题类型}修复
```

**示例**:
- `安全整改: SQL 注入漏洞修复`
- `安全整改: XSS 跨站脚本攻击防护`
- `安全整改: 敏感信息泄露问题修复`
- `安全整改: 身份认证绕过漏洞修复`

### 描述模板

```markdown
## 安全问题概述

**问题类型**: {安全问题类型}
**严重程度**: {严重/中等/低}
**发现时间**: {发现日期}
**影响范围**: {受影响的模块/功能}

## 问题描述

### 漏洞详情
{详细描述安全问题的具体情况，包括漏洞原理、攻击方式等}

### 影响评估
1. **业务影响**
   - {业务影响点1}
   - {业务影响点2}

2. **数据安全**
   - {数据安全风险1}
   - {数据安全风险2}

3. **用户影响**
   - {用户影响点1}
   - {用户影响点2}

## 问题定位

### 涉及文件/代码位置
- {文件路径1:行号} - {问题描述}
- {文件路径2:行号} - {问题描述}
- {文件路径3:行号} - {问题描述}

### 问题根因
{分析问题的根本原因}

## 整改方案

### 修复措施
1. **立即措施**（临时方案）
   - {临时修复措施1}
   - {临时修复措施2}

2. **根本修复**（长期方案）
   - {根本修复措施1}
   - {根本修复措施2}
   - {根本修复措施3}

### 技术实现
- {技术实现方案1}
- {技术实现方案2}

### 安全加固
- {安全加固措施1}
- {安全加固措施2}

## 整改计划

### 时间安排
- **开始时间**: {开始日期}
- **预计完成时间**: {完成日期}
- **上线时间**: {上线日期}

### 责任人
- **开发负责人**: {姓名}
- **安全负责人**: {姓名}
- **测试负责人**: {姓名}

## 验收标准

- [ ] 漏洞已完全修复
- [ ] 通过安全扫描工具检测（无高危漏洞）
- [ ] 通过安全测试用例
- [ ] 代码已通过安全审查
- [ ] 相关文档已更新
- [ ] 已进行安全加固措施
- [ ] 已通知相关团队和用户（如需要）

## 预防措施

为防止类似问题再次发生，建议采取以下措施：

1. **代码审查**
   - 加强安全相关的代码审查
   - 建立安全编码规范检查清单

2. **安全测试**
   - 增加自动化安全测试
   - 定期进行安全扫描

3. **培训教育**
   - 组织安全编码培训
   - 分享安全最佳实践

## 参考信息

- 安全扫描报告: {报告链接}
- 相关安全规范: {规范链接}
- CVE 编号: {CVE编号}（如适用）
- 相关 Issue: #{issue_number}
```

### 使用说明

1. **安全问题分类**: 根据实际问题类型选择合适的标题和描述
2. **严重程度评估**: 准确评估安全问题的严重程度，有助于优先级排序
3. **详细描述**: 详细描述漏洞原理和影响，有助于理解问题的严重性
4. **整改方案**: 提供临时方案和根本方案，确保问题得到彻底解决
5. **时间安排**: 安全问题通常需要紧急处理，合理安排时间节点

## 模板使用指南

### 如何选择合适的模板

1. **识别关键词**: 从用户输入中识别关键词
   - 重构相关 → 使用重构模板
   - 安全相关 → 使用安全问题整改模板

2. **提取信息**: 从用户输入中提取：
   - 模块/组件名称
   - 具体问题描述
   - 相关文件或代码位置
   - 其他关键信息

3. **填充模板**: 将提取的信息填充到模板的占位符中

4. **调整内容**: 根据实际情况调整模板结构，删除不适用的部分，添加必要的信息

### 模板定制建议

1. **项目特定**: 可以根据项目特点添加项目特定的字段
2. **团队规范**: 遵循团队已有的 Issue 格式规范
3. **工具集成**: 考虑与项目管理工具、CI/CD 工具的集成需求

### 注意事项

1. **信息完整性**: 确保填充的信息完整、准确
2. **格式规范**: 保持 Markdown 格式的正确性
3. **可读性**: 使用清晰的标题和列表，提高可读性
4. **可操作性**: 确保描述的内容具体、可执行
